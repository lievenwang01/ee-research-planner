# Workflow

按“模糊想法 → 可执行研究包”的顺序推进。不要一上来就给题目党创新点，也不要在没有问题边界时先堆综述。

## 阶段 1：把模糊意图压缩成任务定义

先判断用户真正要的是什么：
- 开题 / 选题
- 综述 / 文献调研
- 找创新点
- 实验方案
- 吸收某篇或某组论文

然后把用户原话压缩成 5 个槽位：
- 研究对象
- 想解决的任务
- 关键指标
- 关键约束
- 目标产物

**输出**
- 一句“我理解你的问题是……”
- 缺失信息清单（最多 3 个最关键问题）

**停止条件**
- 已经能写出一个不空泛的问题陈述。

## 阶段 2：形成 scoped problem / direction brief

把泛方向变成一个能检索、能比较、能做实验的问题边界。

至少写清：
- 场景 / 系统对象
- 核心问题与为什么重要
- 约束与资源边界
- 预期贡献类型：理论 / 方法 / 工程 / 工具流
- 可达到的验证层级：解析、仿真、benchmark、硬件、HIL

可直接套 `../assets/direction-brief-template.md`。

**输出**
- 1 份 direction brief
- 暂定主赛道 / 副赛道

**检查点**
- brief 里是否出现了“具体对象 + 具体任务 + 具体约束”，而不是只剩热门名词。

## 阶段 3：选择主赛道并限定检索边界

先决定主要贡献承载层，再定义检索边界。

建议同时确定：
- 主赛道 / 副赛道
- 时间范围（例如近 3-5 年 + 经典基线）
- 文献类型（期刊 / 顶会 / benchmark / 标准）
- 证据来源（公开 metadata、OA 全文、已授权访问、用户提供 PDF）

赛道路由不清时，读 `direction-routing.md`。

**输出**
- 一段路由说明：为什么归这个赛道，不归另外两个相邻赛道
- 一组检索边界说明

## 阶段 4：设计 literature plan，而不是只列关键词

把检索拆成 4 组轴：
- 问题轴：对象、场景、任务
- 方法轴：算法、结构、控制律、优化器、架构
- 指标轴：效率、BER、时延、面积、功耗、鲁棒性等
- 约束轴：real-time、edge、low-SNR、high-frequency、resource-limited 等

对每组补：
- 中英文同义词 / 缩写
- 纳入标准与排除标准
- 优先库与全文来源顺序

需要更细的检索法时，读 `literature-strategy.md`。

**输出**
- 关键词分层表
- 检索式草案
- 纳入 / 排除标准
- 优先读的 6-12 篇文献池标准

**检查点**
- 检索计划是否足以区分“同类工作”和“最近邻工作”。

## 阶段 5：把文献整理成 prior-art matrix

不要停在“这篇讲了什么”。把关键文献统一落到卡片和矩阵里。

单篇卡片至少提炼：
- 问题定义
- 方法主干
- 数据 / 仿真 / 硬件平台
- 指标与基线
- 作者声称创新点
- 真正的限制、空白与迁移边界

横向矩阵至少比较：
- 问题边界
- 核心机制
- 关键约束
- 实验平台
- 主要指标
- 明显不足 / 空白

可直接套：
- `../assets/literature-card-template.md`
- `../assets/literature-evidence-pack-template.md`
- `../assets/innovation-matrix-template.md`

需要把 abstract / 网页 / PDF 进一步压成可横向比较的字段时，继续读 `evidence-extraction.md`。

**输出**
- 6-12 篇核心文献的 grouped summary 或矩阵
- 一段“当前赛道最拥挤的位置 / 尚有空间的位置”

**检查点**
- 是否已经能指出最近邻 prior art，而不只是泛泛地说“已有很多工作”。

## 阶段 6：生成 novelty hypotheses

只有在 prior-art matrix 已经显出缺口后，才进入创新假设阶段。

每条假设都要写：
- 对准哪类 prior art 的不足
- 新意来自哪里：问题重构、方法组合、约束改变、评估升级、工程化落地、软硬协同等
- 最近邻参考工作是谁
- 为什么可能有效
- 主要风险是什么
- 最小验证方式是什么

可直接套 `../assets/novelty-hypotheses-template.md`。

**输出**
- 2-4 条候选假设
- 每条假设的风险标签与排序建议

**检查点**
- 每条假设是否都能映射回至少一类具体 prior art 缺口。

## 阶段 7：把假设强制落到 experiment plan

提出创新点时，不要把实验计划留到最后一句。至少要给出一条假设的验证路径。

实验计划至少包括：
- baseline
- 数据 / benchmark / 仿真 / 硬件平台
- 自变量、因变量、控制变量
- 主指标、辅指标、复杂度 / 资源代价
- 消融、鲁棒性、敏感性或失败边界
- 成功 / 失败判据
- 前置资源风险

可直接套 `../assets/experiment-plan-template.md`；需要赛道细节时读 `experiment-design.md`。

**输出**
- 至少 1 条假设对应的实验计划
- 下一步 1 周内能完成的验证动作

**检查点**
- 实验是否真能区分“只是有效”与“相对最近邻工作更值得做”。

## 阶段 8：收束成研究包

默认研究包按下面顺序交付：
1. direction brief
2. 路由判断与关键约束
3. literature plan
4. prior-art matrix / grouped summary
5. novelty hypotheses
6. experiment plan
7. 风险、缺口、下一步

如果用户只要某一段产物，也尽量补两句话：
- 这段产物依赖的上游假设是什么
- 下一段最值得继续做什么

## 接续已有工作

用户不一定从头开始。如果用户说"我已经有……，现在只要……"，先快速定位当前阶段：

| 用户已有 | 直接跳到 | 要补的桥接说明 |
|---|---|---|
| 问题陈述 / brief | 阶段 3（选赛道 + 检索边界） | 先确认 brief 是否已有具体约束 |
| 文献检索计划 | 阶段 5（建 prior-art matrix） | 先问文献池是否已有具体 shortlist |
| prior-art matrix / 综述 | 阶段 6（生成 novelty hypotheses） | 先确认 matrix 是否已经标出了缺口 |
| novelty hypotheses | 阶段 7（实验计划） | 先确认选用哪条假设（conservative/balanced/bold） |
| 实验计划草稿 | 阶段 8（收束成研究包） | 补 validation / reproducibility 短块 |

接续时不要重复已有内容。只补：
- 一句"你已有的产物我理解为……（列出核心结论）"
- 缺失的上游假设或约束
- 当前阶段的输出

## 常见失误

- 没定问题边界就直接列“创新点”。
- 把赛道路由做成关键词匹配，没有看主要贡献层。
- 检索时只列关键词，没有纳入 / 排除标准。
- 只摘 abstract，不区分作者声称创新与真实空白。
- 给出实验计划时没有 baseline、变量或失败判据。
- 在没有全文证据时臆造实现细节。
