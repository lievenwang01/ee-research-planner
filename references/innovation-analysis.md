# Innovation Analysis

目标不是“帮用户编一个创新点”，而是把已有工作拆开，找到仍然值得验证的缺口。

## 1. 先抽取 prior-art 的真实创新单元

对每篇核心论文，分别提炼：
- 问题是如何被限定的
- 方法的关键机制是什么
- 相比基线到底多了什么
- 作者用什么证据证明有效
- 这个创新依赖哪些前提

不要把以下内容误判成创新：
- 只换数据集或只换实验参数
- 仅仅更复杂但没解释为什么有效
- 论文标题写得新，但实验设置比旧工作更宽松
- abstract 里夸大，正文其实只是微调

## 2. 建创新矩阵

矩阵至少要横向比较这些列：
- 问题边界
- 方法机制
- 约束条件
- 实验平台
- 指标改进
- 成本 / 复杂度 / 实时性代价
- 未解决问题

重点不是谁指标最高，而是：
- 谁用了更强假设
- 谁逃掉了难场景
- 谁没有讨论部署成本
- 谁缺少硬件、鲁棒性、极端工况或消融验证

## 3. 生成候选新颖性假设的常用路径

### A. Problem Reframing
把问题定义得更贴近真实约束：
- 加入实时性、功耗、数据缺失、通信受限、器件非理想、极端工况
- 从静态场景改成动态 / 在线 / 分布漂移场景

### B. Mechanism Transfer
把 A 领域有效机制迁移到 B 领域，但要说明兼容条件。

### C. Hybridization
组合两类互补机制，例如：
- 物理模型 + 数据驱动
- 传统控制 + 学习补偿
- 解析算法 + neural refinement
- 架构优化 + EDA 约束反馈

### D. Evaluation Upgrade
方法本身不一定全新，但实验验证层级明显升级：
- 从仿真到 HIL / 硬件
- 从单指标到多目标权衡
- 从理想数据到噪声 / 漂移 / 极端工况

### E. Deployment-aware Design
把部署代价纳入问题本身：
- 存储、算力、带宽、采样率、时延、面积、热、可靠性

## 4. 给每条假设打风险标签

至少标：
- `novelty-risk`: 很可能已被现有工作覆盖
- `evidence-risk`: 需要的数据 / 全文 / baseline 还没拿到
- `resource-risk`: 平台、工具、样机、数据昂贵或不可得
- `scope-risk`: 题目过大，不适合开题或短周期论文

## 5. 优先输出三档候选方案

每次至少给三档：
- `conservative`: 接近已有路线，但更容易做实
- `balanced`: 有一定新意且实验可行
- `bold`: 新意更强，但风险更高

这样用户可以按学位阶段、时间和资源选方案。

## 6. 反证检查

在正式推荐创新点前，再做一次反证：
- 有没有近两年的论文已经做得非常接近。
- 这个假设的贡献，是不是只是换了一个更强模型。
- 如果实验失败，是否还能产出可发表的负结果、边界条件或工程结论。

## 7. 输出最低要求

创新分析阶段至少交付：
- 关键已有创新点列表
- 创新矩阵
- 2-4 条候选假设
- 每条假设的最近邻 prior art
- 每条假设的验证思路和风险
